## 2. P0: 사용자 정보의 소스코드 하드코딩

### 2.1 현상

- 직원·조직·데모용 **사용자 정보**(이름, 사번, 이메일 패턴, 부서, 역할 등)가 **애플리케이션 소스**에 직접 기술되어 있다.
- 일부 화면·로직은 그 **하드코딩된 목록**을 “현재 사용자 매칭”, “조직 트리”, “권한 판단 보조” 등에 사용한다.
- DB(Prisma `User` 등)와 **이중**으로 존재할 수 있어 **어느 쪽이 진실인지** 불명확해질 수 있다.

### 2.2 왜 문제인가

- **운영 반영**: 인사 이동·입퇴사 시 **코드 배포 없이** 반영하기 어렵다.
- **일관성**: 화면/시드/API가 서로 다른 출처를 보면 **불일치·버그**가 난다.
- **보안·개인정보**: 소스에 실명·식별 정보가 쌓이면 **저장소 노출 시 리스크**가 커진다.
- **테스트·다중 환경**: 개발/검증/운영별 사용자 구성을 **코드 분기**로 유지하기 부담스럽다.

### 2.3 목표 상태 (수용 기준)

- **단일 진실 공급원(Single Source of Truth)**: 운영 사용자 정보는 **DB**에만 존재한다. (또는 명시된 **시드/스테이징 전용** 데이터만 예외로 하고, 운영 경로에서는 DB만 사용.)
- 애플리케이션 소스에는 **식별 가능한 개인·조직 목록을 상수로 두지 않는다.**  
  - 예외: **개발 전용** 목업이면 `NODE_ENV` 또는 별도 플래그로 격리하고, 빌드 산출물·운영 배포에 포함되지 않게 한다.
- 화면의 “현재 사용자”, “조직/직원 선택”, API의 “요청 주체”는 모두 **DB `User`(및 조직 정보)** 에서 조회한 결과에만 의존한다.
- 기존 하드코딩 목록을 제거하는 과정에서 **마이그레이션·초기 데이터 반영 계획**(import 또는 시드 일회성)이 문서화되어 있다.

### 2.4 관련 작업 (하위 요구사항과 연결)

- [R-AUTH] 인증 도입 후, “누구인지”는 **세션/JWT + DB User** 로만 결정.
- [R-ADMIN-IMPORT] 관리자 **CSV/Excel import** 로 대량 반영 가능.
- [R-ADMIN-GRANT] 관리자 화면에서 **역할·소속 등 권한에 해당하는 필드**를 부여·변경.
- [R-UI-MIGRATE] **권한·조직·시스템에 영향을 주는 수정 UI**는 관리자 화면으로 이관.
- [R-CONFIG] 도메인 규칙 등 **비개인 설정**만 env/설정 테이블로 분리 (개인정보는 소스에 두지 않음).
- [R-ORG-CODES] 부서·센터·직책·직급 **표기**는 코드 사전(§10)으로 DB 관리 가능하게 한다.

## 3. 인증·식별 (R-AUTH)

### 3.1 요구사항

- 로그인은 **서버가 검증**한다 (비밀번호 또는 사내 SSO 등; 구체 방식은 합의).
- 로그인 성공 후 **세션(쿠키)** 또는 **JWT**로 이후 요청의 주체를 증명한다.
- API는 **`?email=` 등 클라이언트 임의 문자열을 신뢰 원천으로 사용하지 않는다.**

### 3.2 수용 기준

- 인증 없이 타인 데이터를 조회·변경할 수 있는 경로가 **없다** (또는 개발 전용으로 완전 차단).
- 로그아웃·세션 만료 동작이 정의되어 있다.

## 4. 사용자 키 설계 (R-KEYS)

### 4.1 요구사항

- **내부 PK**: `User.id` — 모든 FK·조인의 기준.
- **선택**: `public_id` (UUID) — URL·토큰 `sub` 등 외부 노출 최소화.
- **업무 키**: `employeeNo`(사번) — HR 연동·import 매핑; 보안은 **난수 ID가 아니라 인증**으로 확보.
- **이메일**: 로그인 ID·연락용으로 사용 가능; 변경 시에도 `User.id` 기준 권한이 유지되도록 설계.

### 4.2 수용 기준

- 권한 판단 입력은 항상 **인증으로 확인된 `User` 레코드**에서 온다.
- 마이그레이션: 기존 행에 `public_id` 도입 시 **백필 규칙**이 있다.

---

## 5. 권한 모델·적용 (R-PERM)

### 5.1 요구사항

- 업무 권한은 DB의 **`User.role`(`OrgRole`)**, **`dept`**, **`subDept`**(및 필요 시 등급 필드)와, 목표 레벨·데이터 소유 관계 등 **기존 규칙**을 기반으로 판단한다.
- **권한에 해당하는 값의 변경(부여·회수·이동)** 은 **관리자만** 수행한다. ([R-ADMIN-GRANT], [R-UI-MIGRATE]와 연계)
- “어떤 User로 요청하는가”는 **오직 인증 결과**에서만 결정한다.

### 5.2 수용 기준

- 다른 사용자의 이메일·사번을 조작하여 권한 우회가 **불가능**하다.
- 일반 사용자 UI에서 **타인의 역할·소속·관리자 플래그**를 바꿀 수 있는 경로가 **없다**.

## 6. 일반 화면 vs 관리자 화면 (R-UI-MIGRATE)

### 6.1 원칙

- **일반 사용자 화면**: 본인의 KPI 업무(목표·점검·실적·알림 등) **조회·입력** 중심. **조직 전체·타인 계정·시스템 정책을 바꾸는 수정**은 두지 않는다.
- **관리자 화면**: 사용자·권한·조직 연동·시스템 설정 등 **운영·거버넌스** 성격의 **조회·수정·일괄 반영**을 전담한다.
- 현재 일반 화면(또는 API)에 존재하는 **관리 성격의 수정 기능**은 식별하여 **관리자 화면(및 관리자 전용 API)** 으로 **이관**한다. 이관 후 일반 경로에서는 해당 쓰기 API/UI를 **제거하거나 비활성화**한다.

### 6.2 이관 대상 (판단 기준)

다음에 해당하면 **관리자 전용**으로 옮긴다 (구체 화면 목록은 구현 전 **현행 코드 갭 분석**으로 확정).

- **다른 사용자**의 계정·역할·부서·소속 센터를 바꾸는 기능
- **전사·본부·센터 단위 정책**(점검 주기, 시스템 설정, 공통 코드 등)을 바꾸는 기능
- **감사·보안**상 운영자만 다뤄야 하는 설정

### 6.3 수용 기준

- 관리자가 아닌 사용자로 로그인했을 때, 위 이관 대상에 해당하는 **페이지·버튼·API 쓰기**에 접근할 수 **없다**.
- 관리자 화면은 [R-ADMIN-SEC]를 만족한다 (역할 + IP 등).

## 7. 관리자 기능 (R-ADMIN)

### 7.1 권한 부여·변경 (R-ADMIN-GRANT)

- 관리자는 지정 사용자에 대해 다음을 **화면에서 조회·수정**할 수 있다 (필드는 DB 스키마와 일치하도록 합의).
  - **역할(`OrgRole`)** 부여·변경
  - **본부(`dept`)·센터(`subDept`)** 등 소속 정보
  - (필요 시) **등급(`gradeLevel`)** 및 기타 권한 판단에 쓰이는 속성
- **시스템 최고 관리자**와 **일반 관리자** 등 **계층**이 필요하면 역할을 세분화한다 (별도 enum 또는 플래그; 합의 사항).
- 모든 변경은 **감사 로그**(대상 사용자, 변경 전·후, 실행 관리자, 시각)에 남긴다.

**수용 기준**

- 권한 변경은 **관리자 인증 + 관리자 권한 + (정책 시) IP 제한**을 모두 통과한 경우에만 가능하다.
- Import([R-ADMIN-IMPORT])로 일괄 반영된 내용과 **충돌 없이** 병행 가능한 규칙(마지막 반영 우선 등)을 문서화한다.

### 7.2 비밀번호 초기화 (R-ADMIN-PW)

- 관리자가 지정 사용자의 비밀번호를 **재설정**할 수 있다.
- 임시 비밀번호·첫 로그인 변경 강제 등 **정책**을 합의한다.

### 7.3 사용자 Import (R-ADMIN-IMPORT)

- **Excel 또는 CSV** 업로드로 `User`(및 필요 시 조직·역할 필드)를 DB에 반영한다.
- 사번·이메일 등 **키 충돌 시 규칙**(신규/갱신/스킵)을 정의한다.
- **실패 행 리포트**; 가능하면 **미리보기(드라이런)**.
- **감사 로그**(실행자·시각·건수).

### 7.4 관리자 보안 (R-ADMIN-SEC)

- 관리자 전용 경로에 **IP 허용 목록**(사내망)을 둔다.
- **관리자 역할**과 일반 `OrgRole` 기반 업무 권한을 **분리**해 정의한다 (동일 인물이 둘 다 가질 수 있는지는 합의).
- (선택) MFA 또는 SSO.

### 7.5 회원가입

- 기본안: **공개 회원가입 없음**; 계정은 import 또는 관리자 생성으로만 반영.  
- 추후 필요 시 “초대 기반 가입” 등은 **별도 요구사항**으로 분리.

## 8. 설정·하드코딩 완화 (R-CONFIG)

### 8.1 요구사항

- **개인·식별 정보**는 소스에 두지 않는다 (§2와 동일 원칙).
- **도메인 허용 규칙**, 기본 연도 등 **비개인 정책**은 환경 변수, DB 설정 테이블, 또는 **관리자 화면에서 수정 가능한 설정**(§6·§7과 정합)으로 이전한다.

### 8.2 수용 기준

- 운영자가 “설정 변경”에 **재배포가 필수가 아닌** 항목이 문서에 구분되어 있다.

## 9. 비기능·운영 (R-NFR)

- **HTTPS** 배포 전제(세션 쿠키 시 특히).
- **백업**: import·권한 일괄 변경 전 스냅샷 또는 트랜잭션 정책 검토.
- **문서**: 배포·설정·관리자 절차를 README 또는 운영 가이드에 요약.

## 10. 조직·직책·직급 코드 사전 (R-ORG-CODES)

`dept`, `subDept`, 화면에 보이는 **역할(role)·직급(grade)** 표기를 **고정 문자열이 아니라 코드**로 두고, **한글(또는 표시명)은 DB에서만** 바꿀 수 있게 한다.

**권한 판별**은 기존과 같이 `User.role`(`OrgRole` enum), `User.gradeLevel`(`GradeLevel` enum)을 사용한다.  
코드 사전은 **표시명·콤보 목록·CSV/리포트 라벨**을 DB로 운영하기 위한 레이어이며, 필요 시 이후 `User.deptCode` 등 FK로 완전 코드 기반 전환이 가능하다.

### 10.1 테이블 요약

| 모델 | 용도 | PK `code` 예시 | 한글 예시 (`labelKo`) |
|------|------|----------------|------------------------|
| `OrgDeptCode` | 본부·기관 단위 | `dept_inst` | 기관장 |
| `OrgSubDeptCode` | 센터·실·팀 | `sd_office_pres` | 원장실 |
| `OrgRoleCode` | 직책 표시 | `role_president` | 원장 |
| `OrgGradeCode` | 직급 표시 | `grade_lv1_ga` | 1급갑 |

공통: `sortOrder`, `active`, `createdAt` / `updatedAt`.

### 10.2 본부 코드 (`OrgDeptCode`)

- 코드 규칙: `dept_` + 식별자, 또는 `dept01`·`dept02` 등 **숫자 스타일**도 가능. **PK `code`만 유일**하면 된다.
- `labelKo`는 당분간 `User.dept` 문자열과 **맞추면** 마이그레이션이 쉽다.

**패턴 예시**

| code | labelKo |
|------|---------|
| `dept01` | 기관장 |
| `dept02` | 미래성장전략본부 |
| `dept03` | 경영기획본부 |

**시드에서 쓸 수 있는 예시** (`dept_inst`, `dept_future`, `dept_plan`, `dept_green`, `dept_energy`, `dept_std`, `dept_common` 등)는 구현 시 시드와 본 표를 **동기화**할 것.

### 10.3 센터·하위조직 (`OrgSubDeptCode`)

- PK 예: `sd_` + 구분자. 반드시 상위 `deptCode`로 `OrgDeptCode`와 연결.
- 동일 `labelKo`라도 본부가 다르면 **코드는 다르게** 두는 것을 권장.

**예시**

| code | deptCode | labelKo |
|------|----------|---------|
| `sd_office_pres` | `dept_inst` | 원장실 |
| `sd_office_vp` | `dept_inst` | 부원장실 |
| `sd_hq_default` | `dept_future` | 본부직할 |
| `sd_climate` | `dept_future` | 기후변화인증센터 |
| `sd_dx` | `dept_future` | DX인증센터 |
| `sd_global` | `dept_future` | 글로벌사업센터 |
| `sd_plan_center` | `dept_plan` | 경영기획센터 |
| `sd_leave` | `dept_common` | 휴직 |

### 10.4 직책 코드 (`OrgRoleCode`)

- `permissionKey` (nullable, `OrgRole`): **있으면** 시스템 권한과 1:1 대응(동일 enum 중복 행 지양). **NULL이면** 표시 전용 직함(예: 수습, 실장); 권한 연동 시 정책 별도 정의.

**예시**

| code | labelKo | permissionKey |
|------|---------|----------------|
| `role_president` | 원장 | `PRESIDENT` |
| `role_vp` | 부원장 | `VICE_PRESIDENT` |
| `role_hq_head` | 본부장 | `HQ_HEAD` |
| `role_center_head` | 센터장 | `CENTER_HEAD` |
| `role_staff` | 직원 | `STAFF` |
| `role_intern` | 수습 | *(NULL)* |

### 10.5 직급 코드 (`OrgGradeCode`)

- `mapsToGrade` (nullable, `GradeLevel`): **있으면** 표기 → 내부 단(LEVEL_1~4) 매핑. 동일 `GradeLevel`에 여러 `labelKo` 가능.
- **NULL**이면 enum 밖 표기용.

**예시**

| code | labelKo | mapsToGrade |
|------|---------|-------------|
| `grade_lv1_ga` | 1급갑 | `LEVEL_1` |
| `grade_lv1_eul` | 1급을 | `LEVEL_1` |
| `grade_lv2_ga` | 2급갑 | `LEVEL_2` |
| `grade_lv2_eul` | 2급을 | `LEVEL_2` |
| `grade_lv3_ga` | 3급갑 | `LEVEL_3` |
| `grade_lv3_eul` | 3급을 | `LEVEL_3` |
| `grade_lv3_byeong` | 3급병 | `LEVEL_3` |
| `grade_lv4_ga` | 4급갑 | `LEVEL_4` |
| `grade_lv4_eul` | 4급을 | `LEVEL_4` |
| `grade_custom` | 특임 | *(NULL)* |

### 10.6 `User`와의 매칭 (현재 / 향후)

| User 필드 | 코드 사전에서의 활용 |
|-----------|----------------------|
| `dept` | `OrgDeptCode.labelKo` 일치 → `code` (또는 향후 `deptCode` FK) |
| `subDept` | 상위 `deptCode` + `labelKo`로 `OrgSubDeptCode` 검색 |
| `role` | `OrgRoleCode.permissionKey = User.role` → 화면은 `labelKo` |
| `gradeLevel` | `OrgGradeCode.mapsToGrade` 일치 행 중 선택 (`sortOrder` 등으로 단일화) |

**향후(선택):** `User.deptCode`, `subDeptCode`, `roleCode`, `gradeCode` FK로 표기 변경을 `labelKo` 수정만으로 처리.

### 10.7 운영 체크리스트 (신규 코드)

1. 신규 본부 → `OrgDeptCode` 행 추가, `active=true`.  
2. 신규 센터 → `OrgSubDeptCode`에 `deptCode`·`code`·`labelKo` 추가.  
3. 직함만(권한 없음) → `OrgRoleCode`에 `permissionKey=NULL` 후 정책 확정 시 enum 연동.  
4. 직급 표기만 → `OrgGradeCode`에 `mapsToGrade` 선택적 연결.  
5. 문자열 기반 `User`를 쓰는 동안 `dept`/`subDept`는 새 `labelKo`와 일치시킨다.

### 10.8 구현·배포 시

- 스키마에 위 네 엔티티를 둔다. 초기 데이터는 시드 또는 마이그레이션으로 넣는다.  
- 관리/드롭다운 API는 `code`·`labelKo`를 내려주고, 리포트·CSV는 **코드 저장 + 출력 시 라벨 조인**을 권장.

6. **P2** — [R-ADMIN-IMPORT]·[R-ADMIN-PW]·관리자 UI  
7. **P2** — [R-CONFIG] 비개인 설정의 env/DB/관리자 설정 이전  
8. **P2** — [R-ORG-CODES] 코드 사전과 화면·import·리포트 연동 (표시 계층부터 적용 가능)
